Flaky Shop × flakehound a living dashboard — real site, planted bugs, clustered failures

How this works — a three-stage chain

  1. 1 · The Flaky Shop is a real website with bugs planted on purpose. A tiny login → checkout → payment storefront (deployed on Vercel) with three defects in the application code: a pay button that renders after a random delay, a cart total that races an async promo, and a receipt total that is consistently miscalculated.
  2. 2 · A Playwright suite runs against the live site every few hours. Each run emits standard JUnit XML plus a metadata sidecar (commit SHA, timestamp, runner) into a dated history folder. The two timing bugs surface as intermittent failures; the miscalculated receipt fails every run.
  3. 3 · flakehound analyzes the accumulated history. Flakiness is scored by pass↔fail transition frequency on the same commit (not naive fail rate); the consistently-broken receipt is separated out as a regression; and failures are clustered by normalized stack trace. This runs with --no-ai — the deterministic core, free forever. This page renders the resulting flakehound.report.json.

Honesty, in full: the flakiness here is seeded on purpose for demonstration — the bugs live in the app, not the tests. flakehound analyzes test artifacts (JUnit XML); it never inspects the running website. The oldest runs in the history are a hand-written seed backstory that bootstraps the baseline so the flaky-vs-regression split is visible from the first render; every run after that is a genuine artifact from the deployed site. Clustering knows stack traces, not source code — it groups failures by shared cause and, for regressions, names the commit that broke them.
