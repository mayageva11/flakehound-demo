Flaky Shop × flakehound living demo — real site, planted bugs

How this works — A three-stage analysis chain

  1. 1
    The Flaky Shop — a real site with bugs planted on purpose
    A tiny login → checkout → payment storefront (served from this repo's GitHub Pages at /shop/) with three defects in the application code: a pay button that renders after a random delay, a cart total that races an async promo, and a receipt total that is consistently miscalculated.
  2. 2
    A Playwright suite runs against the live site every few hours
    Each run emits standard JUnit XML plus a metadata sidecar (commit SHA, timestamp, runner) into a dated history folder. Retries are enabled in CI, so a fail-then-pass within one run — flakehound's highest-confidence flaky signal — lands in the artifacts too.
  3. 3
    Analysis by the flakehound CLI
    Flakiness is scored by pass↔fail transition frequency on the same commit (not naive fail rates); the consistently-broken receipt is separated out as a regression; failures are clustered by normalized stack trace with head-weighted similarity. This page renders the resulting flakehound.report.json.
Honesty, in full: the flakiness here is seeded on purpose for demonstration — the bugs live in the app, never in the tests. flakehound analyzes test artifacts (JUnit XML); it never inspects the running website. The oldest runs in the history are a hand-written, clearly-labelled seed backstory that bootstraps the baseline; every run after that is a genuine artifact from the deployed site. AI hypotheses (generated by a local model at zero cost) are interpretive additions and never alter the deterministic scores, clusters, or gate verdicts.
